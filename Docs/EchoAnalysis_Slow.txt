  Executive Summary

  Your application is experiencing severe CPU and performance issues due to excessive main thread activity caused by the
  streaming query architecture. The system is triggering thousands of SwiftUI view updates during query execution,

  Why Application is Sluggish

  During Query Execution:

  1. PostgreSQL fetch loop (background thread) is doing triple data processing per cell
  2. ResultStreamBatchWorker flushes batches every 20-450ms
  3. Each batch triggers MainActor update in applyStreamUpdate()
  4. This triggers WorkspaceTab notification cascade
  5. 8 onChange modifiers in QueryResultsSection fire
  6. Some trigger row reordering (sorting 100K indices)
  7. Stats emitted 200x/second during preview
  8. File I/O on MainActor for metadata/stats persistence

  After Query Completes:

  1. Row cache misses trigger disk I/O during scrolling
  2. Multiple data copies cause memory pressure
  3. Background GC tries to clean up redundant allocations
  4. SwiftUI continuously recomputes views due to change notifications

  ---
  Root Cause Analysis

  The fundamental issue is architectural:

  1. Overly granular updates: System designed for real-time streaming UI, but creates too many notifications
  2. Main thread bottleneck: All state updates forced to MainActor, blocking on I/O and processing
  3. Redundant work: Data copied and transformed multiple times
  4. Reactive cascade: SwiftUI's reactivity amplifies every small change into expensive view recomputations
  5. Premature optimization: Complex spool/cache system adds overhead without clear benefit for 100K row dataset

  ---
  Why This Gets Worse With Larger Datasets

  - Quadratic complexity: More rows = more updates × more expensive operations per update
  - Memory pressure: 4x data copies means earlier GC thrashing
  - I/O bottleneck: More spool operations on main thread
  - View complexity: Sorting 1M rows takes ~100x longer than 10K rows

  ---
  Recommendations (No Code Changes)

  To achieve sub-12s for 100K rows, you need to:

  1. Eliminate main thread file I/O - Move persistMetadata/persistStats off MainActor
  2. Reduce notification frequency - Batch updates or use dedicated update cadence (e.g., 250ms)
  3. Simplify data processing - Pick one format (either preview strings OR binary) during fetch
  4. Defer expensive operations - Don't rebuild row order during streaming
  5. Reduce data copies - Use single source of truth for row storage
  6. Optimize batch sizes - Larger batches = fewer updates = less overhead
  7. Lazy view updates - Only recompute visible cells
  8. Profile with Instruments - Confirm hotspots with Time Profiler and System Trace

  The current architecture optimizes for low latency to first rows but sacrifices overall throughput. For your 100K row
  goal, you need to rebalance toward throughput.
