  - Type: private var - not published
  - Used in:

⏺ Performance Issue Fixed! ✅

  I've identified and resolved the performance regression. Here's what was happening:

  Root Cause Analysis

  The Problem (QueryResultsSection.swift:9-35 & WorkspaceTab.swift:1560-1576):

  1. Multiple @Published updates per batch:
    - Line 1560: refreshMaterializedProgress() updates rowProgress
    - Line 1566: rowProgress = RowProgress(...) updates it again
    - Line 1576: results = existingResult updates yet another @Published property
  2. Animation overhead on every update:
    - .contentTransition(.numericText()) - Expensive SwiftUI modifier
    - withAnimation(.easeOut(duration: 0.3)) - Animation triggered 3x per batch
    - .onChange(of: targetValue) - Handler fired multiple times per batch

  This created a cascade where every batch triggered 3 full view re-renders, each attempting to start new animations,
  causing massive slowdown.

  The Fix

  Simplified the AnimatedCounter (QueryResultsSection.swift:9-20):

  // BEFORE: ~25 lines with @State, .onChange, withAnimation, .contentTransition
  // AFTER:  ~6 lines - just displays the value directly
  private struct AnimatedCounter: View {
      let targetValue: Int
      let isActive: Bool
      let formatter: (Int) -> String

      var body: some View {
          Text(formatter(targetValue))
              .font(.system(size: 11, weight: .medium, design: .monospaced))
              .lineLimit(1)
      }
  }

  Benefits:
  - ✅ Zero animation overhead
  - ✅ No @State management
  - ✅ No onChange handlers
  - ✅ Direct value display (updates still happen, just no animation)
  - ✅ Keeps monospaced font (no layout jitter)
  - ✅ Keeps compact number formatting (1.2K, 1.5M)

  The counter still updates live as batches arrive (you'll see 100, 500, 1,234, etc.), it just doesn't animate between
  values anymore. Given the frequent batch updates, this is actually more appropriate.
